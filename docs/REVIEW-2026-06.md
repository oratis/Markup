## 总体结论

完成度与工程纪律都明显高于同类个人开源项目。0.7.0 已签名公证、带签名自动更新；桌面端功能基本是 Typora 与 Obsidian 的并集；iOS 原生 reader 也已进 TestFlight。约 96 个前端测试文件、Rust 单元+集成测试、iOS 逻辑层镜像测试，文档齐全。工程上距 1.0 几乎没有硬缺口——唯一历史"硬阻塞"是上架方式的商业决策，已在 ADR-003 澄清（1.0 直分发优先，MAS 1.0 后评估）。

两类问题建议在继续加功能前先处理：

1. **真实的安全/越权问题**：后端 `read_file` / `write_file` / `rename_file` 只校验扩展名、不校验路径范围。
2. **可维护性债**：`App.tsx` 单文件 3,200+ 行、零测试；组件层测试覆盖约 40%。

最大风险不在技术，而在**产品定位**：同时对标 Typora、Obsidian、GitHub 文档阅读器，身份模糊。详见 PRODUCT-DIRECTION。

## 1. 桌面端（macOS / Tauri 2 + React 19 + Zustand + Milkdown + CodeMirror 6）

### 功能盘点（已相当完整，多数打磨到位）

- 三模式（阅读 / WYSIWYG / 源码）一键切换，CJK 输入法不闪烁
- Vault 文件树（虚拟滚动）、多标签（pin、拖拽重排）、会话恢复、按标签记忆滚动位置
- Quick Open（`⌘P`，含 `#`标题 / `^`块 模式）、命令面板（`⌘⇧P`）、全文搜索（`⌘⇧F`，`tag:` / `path:` 操作符，Tantivy）
- Wikilinks、反链面板、力导向图谱、大纲（>50KB 用 worker）、标签面板、书签
- Canvas（`.canvas` 白板，与 Obsidian 格式互通，撤销/重做、节点/连线/分组）
- 导出 HTML / 打印 PDF（GitHub / plain / Tufte 三主题，保真 KaTeX、Mermaid、代码高亮）
- GitHub 设备流登录 + 仓库浏览、frontmatter 属性编辑器、图片粘贴入库、专注/打字机模式、快捷键编辑器、主题、双语 UI、首启引导、被动更新提示

### 架构

状态集中在 [store.ts](../src/store.ts)（约 717 行）+ 四个并行索引 store（链接/标签/标题/块），Canvas 每标签一个独立 store。复杂度高度集中在 [App.tsx](../src/App.tsx)——3,200 多行，命令注册、键盘处理、副作用编排、弹窗状态都在一个文件里。**当前最大的结构性隐患**：难测试、改一处易牵连多处。

### 测试与质量

业务逻辑（store、lib 纯函数、Canvas）覆盖很好；40 个组件只有 16 个有测试，`App.tsx`、`Editor`、`SourceEditor`、`FileTree`、`SettingsDialog`、`GraphView` 等关键组件无测试。可访问性（自定义控件缺 ARIA）、大 vault 索引重建无进度反馈是两处用户可感知短板。Find Bar 在 WYSIWYG 下正则不能逐项跳转，属已知不完整。

## 2. Rust 后端（`src-tauri/`，约 3,100 行）

整体扎实：20 个命令按域分模块；写文件是**临时文件 + 原子 rename + mtime 冲突检测**；HTML 导出经 **ammonia 白名单净化**，XSS 防护到位；HtmlView 预览 iframe 明确不给 `allow-scripts`；更新走 **minisign 验签**。

需关注：

- 🔴 **路径范围缺失（中等严重）**：`read_file` / `write_file` / `rename_file` 接收任意绝对路径，只判扩展名（[commands.rs:86](../src-tauri/src/commands.rs) 起）。前端（或注入到 webview 的代码）可读写系统上任意 `.md` / `.html` / `.canvas` 文件，越出当前 vault。对照 `write_image` 已做 `..`/绝对路径/盘符完整防护——同样校验应补到这三个命令。
- 🟠 **GitHub token 存 `localStorage`**（[github-auth.ts:14](../src/lib/github-auth.ts)）而非 Keychain；只读 scope、风险有限，但明文易被任意脚本读取。
- 🟠 mtime 冲突检测 1.5s 容差，双窗口并发编辑同一文件可能静默覆盖；vault 每次打开**全量重建索引**，更大库会卡。

## 3. iOS（原生 SwiftUI，非 RN/Expo）

定位是 macOS 版的 reader-first 伴侣，亮点是**把任意 GitHub 仓库当 vault 打开、离线阅读**——iOS Markdown 阅读器里少见。流程：zipball 下载 → 原子解压 → 扫描文档引用的图片/CSS 一并下载 → 交给本地 vault 管线（FTS5 索引）；"Pull latest" 用 manifest SHA 差量只下改动文件。

成熟度：M0–M4（阅读、导航、轻量编辑、分享导出、双语、无障碍、iPad 多标签+分屏）完成；G0–G6（GitHub 集成）大部分已发。逻辑层 MarkupKit 把桌面 `fuzzy` / `search-operators` / `wikilink` 等**重新移植成 Swift**（非共享代码），配镜像测试。**缺口**：无图谱、Canvas 仅占位、编辑为轻量源码非 WYSIWYG、GitHub 仓库只读、跨文档 `#fragment` 锚点滚动未做。MarkupApp（UI 层）无单测，靠 CI 编译检查 + 手测。

## 4. 基础设施 / CI / 文档

CI/CD 健全：PR 跑 Frontend（Biome + tsc + vitest + 覆盖率）、Rust（cargo test + llvm-cov）、E2E（Playwright）、iOS（swift test + 模拟器编译）；release 工作流做**双架构构建 + Developer ID 签名 + 公证 + 装订 + 签名 `latest.json` 更新 feed**。版本在 `package.json` / `tauri.conf.json` / `Cargo.toml` 三处一致为 0.7.0。文档极完整。

短板：覆盖率上传 Codecov 但**无失败门槛**；**38 条远端分支**多为 dependabot，积压带来 rebase 成本且无自动合并策略；公证步骤**无重试**，Apple 服务慢会让 release 在 15 分钟超时内挂掉；`RELEASE-PLAN.md` 头部措辞轻微过期。
