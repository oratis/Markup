## 1. 现状定位与核心矛盾

当前一句话定位：**"快速、原生、开源的 macOS Markdown 编辑器——像漂亮网页一样阅读，想编辑就编辑。"**

问题在于产品实际在追三个不同身份，目标用户与留存逻辑差别很大：

1. **Typora 式** reader-first 单文件编辑器
2. **Obsidian 式** 知识库（vault + 双链 + 图谱 + Canvas）
3. **GitHub 文档阅读器**（iOS 主打）

身份分散会稀释叙事，也分散有限的开发精力。

## 2. 竞争现实

- **vs Typora**：$15 一次性、reader-first 打磨多年。Markup 优势是免费/开源/原生 + vault——够，但非碾压。
- **vs Obsidian**：护城河是**社区插件生态 + 跨平台同步**，不是图谱/双链本身。本项目把"插件/同步"列为 out-of-scope，意味着**正面打 Obsidian 是打不赢的消耗战**。
- **vs 系统/免费工具**：macOS 上免费 Markdown 编辑器众多，"又一个编辑器"难被记住。

## 3. 推荐定位：锚定"阅读 + GitHub 原生"

把定位收敛到别人没占的象限：**阅读器优先 × 知识库**。

```
                    知识库 / vault
                          │
            Obsidian ●    │    ● Markup  ← 空白象限（推荐锚点）
                          │
  编辑器优先 ─────────────┼───────────────── 阅读器优先
                          │
   iA · MacDown ●         │
              Typora ●    │
                          │
                       单文件
```

- Typora / iA / MacDown 在"编辑器优先 × 单文件"
- Obsidian 在"编辑器优先 × 知识库"
- **Markup 是唯一把渲染当产品、而非编辑预览的**，独占"阅读器优先 × 知识库"

两条放大理由：

1. **"reader-first" 是真正独特的姿态**——一句话讲得清、竞品学不像。
2. **GitHub 文档阅读是被低估的杀手场景**——开发者/技术写作者海量 Markdown 躺在 GitHub（README、docs、wiki、笔记），今天没有好的原生阅读器。iOS 已验证方向，桌面端却只做了单向 fetch。

新定位表述（建议）：**"Markdown 文档的原生阅读器 + 轻量编辑器，跨 Mac/iOS，深度集成 GitHub。"**

收敛 ≠ 砍功能，而是让叙事和路线图围绕"阅读"与"GitHub"排序，把 Canvas/图谱作为"知识库能力"的支撑，而非主打去和 Obsidian 硬碰。

## 4. 分层路线图

### A 层 · 先还债（发 1.0 前，本轮推进）

| 项 | 内容 | 优先级 |
|---|---|---|
| 后端路径校验 | 三个文件命令限制在 vault 范围 + 测试 | P0 |
| 拆 `App.tsx` | 按域抽 hooks，降复杂度、可测 | P1 |
| 索引进度 | 大 vault 索引进行中反馈 | P1 |
| token 入 Keychain | Rust 侧 Keychain 存取，前端不留明文 | P2 |
| 测试与门槛 | 补关键组件测试 + CI 覆盖率底线 | P2 |
| 杂项清理 | dependabot 批处理、公证重试、Find Bar 正则、帮助页 | P3 |

### B 层 · 强化差异化（1.x，把"阅读 + GitHub"做深）

- **桌面端 GitHub 双向化**：现仅拉取，补"编辑副本 → 提交 PR / fork"，从只读阅读器升级为 GitHub 文档轻量协作端。**与所有竞品拉开差距的点。** → 设计已就绪：[design/06-github-roundtrip.md](design/06-github-roundtrip.md)（批次 B301–B307）。
- **文档站点式阅读**：把一个仓库/文件夹当站点来读——侧栏目录树 + 全局 TOC + 跨文档面包屑 + `#fragment` 锚点跳转（iOS 已记待办，桌面也值得）。让"读一整套文档"成为强项。
- **Mac ↔ iOS 体验对齐**：统一 GitHub vault 的打开/刷新/缓存模型；reader 设置（字号/行宽/主题）两端对齐。
- **整站导出**：多文档 + 资源打包成可离线 HTML，强化"一键把 Markdown 变成可分享网页"。

### C 层 · 战略与增长

- 定 1.0 分发：直分发优先发 1.0，MAS 作为 1.0 后触达增量（见 ADR-003）。
- 可持续性：纯免费 MIT 可行；若长期投入，考虑"App 免费开源 + 可选同步/团队服务收费"，避免走 Obsidian 免费插件生态老路。
- 增长抓手围绕新定位：把"在 Mac/iOS 上原生阅读任意 GitHub 仓库文档"做成可演示 hero 场景。

## 5. 三项决策（推荐）

| # | 决策 | 推荐 |
|---|---|---|
| 1 | 1.0 分发 | **直分发优先发 1.0**，MAS 后续增量 |

| 3 | 主打叙事 | **立"GitHub 文档原生阅读器"为主打**，含桌面 GitHub 双向化投入 |
